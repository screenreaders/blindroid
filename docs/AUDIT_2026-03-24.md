## Architecture Overview

### Modules
- `app/`: Blindroid main application (UI hub, onboarding, diagnostics, call/sms, navigation, scanners).
- `launcher/`: Custom launcher.
- `braillekeyboard/`: Braille IME module.
- `external/talkback/`: BlindReader (TalkBack-based screen reader), settings, gestures, AI/OCR, plugins.

### Runtime Flow (BlindReader)
1. Android Accessibility events are received by `TalkBackService`.
2. `AccessibilityEventProcessor` routes events to interpreters (focus, text, scroll, screen state).
3. Interpreters emit feedback into `Pipeline`.
4. `Pipeline` invokes `Actors` (speech, focus, actions, UI overlays, OCR/AI).
5. `SpeechControllerImpl` and `Compositor` produce spoken/haptic feedback.

Key files:
- `external/talkback/talkback/src/main/java/com/google/android/accessibility/talkback/TalkBackService.java`
- `external/talkback/talkback/src/main/java/com/google/android/accessibility/talkback/Pipeline.java`
- `external/talkback/talkback/src/main/java/com/google/android/accessibility/talkback/actor/*`

### Runtime Flow (Blindroid app)
- `MainActivity` is the hub for user-facing modules and navigation.
- `OnboardingActivity` + `.accessibility.TalkbackWizardActivity` guide first-run configuration.
- Services: notification listener, call service, navigation foreground services.
- Assist activities (obstacle/face/currency/document/scan) support camera-based recognition.

Key files:
- `app/src/main/java/com/screenreaders/blindroid/MainActivity.kt`
- `app/src/main/java/com/screenreaders/blindroid/OnboardingActivity.kt`
- `app/src/main/java/com/screenreaders/blindroid/DiagnosticsActivity.kt`

## iOS Gesture Parity (from `info/ios.txt`)
Current iOS gesture set defaults exist, but parity is **not complete**. Gaps:

1. **Rotor gesture (two-finger rotate)**
   - Implemented as rotate cw/ccw → next/previous reading setting (rotor-like).

2. **Split tap (hold one finger + tap another)**
   - No explicit mapping found; TalkBack supports pass-through but not split-tap semantic.

3. **Scrub gesture (2-finger “Z”)**
   - Not mapped (Back/Close via scrub is missing).

4. **3‑finger single tap (extra info)**
   - Updated to announce item position (extra info).

5. **4‑finger top/bottom semantics**
   - iOS distinguishes top/bottom screen. Current mapping is single/double tap; lacks top/bottom specificity.

6. **Item chooser gesture**
   - iOS uses 2‑finger triple tap; current mapping is screen search (close but not identical).

7. **Quick settings gesture**
   - iOS uses 2‑finger 4‑tap; current mapping is 4‑finger 2‑tap‑hold.

## BlindReader Gaps (Functional)
- **iOS gesture parity gaps** listed above.
- **AI assistant actions**: Gemini voice command currently forwards prompt and returns response; it does not execute multi-step actions or system commands beyond existing voice command set.
- **Live recognition**: image description is available; continuous/live recognition mode (like VoiceOver “Live Recognition”) is not explicitly exposed as a persistent mode.

## Blindroid App Gaps / Testability
- Several internal activities (`OnboardingActivity`, `DiagnosticsActivity`, `TalkbackWizardActivity`) are not exported, which blocks shell-based automated functional tests.
- No dedicated “test mode” or instrumentation hooks for launching non-exported flows.

## Recommendations (Next Steps)
1. Add iOS rotor + scrub + split‑tap support and expose them in the iOS gesture set.
2. Provide explicit Item Chooser / Quick Settings mappings for iOS parity.
3. Add optional continuous/live recognition mode for BlindReader.
4. Add a test-only activity/intent flag to allow automated functional testing of onboarding/diagnostics flows.
